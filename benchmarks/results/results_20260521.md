> **Why the embedded string is wrong:** Vim embeds its Makefile CFLAGS into

> to overwrite them. The shim overwrites `CFLAGS` at compile time, so the
> binary is actually built with `-Os -march=armv8-a`, but `strings $(which vim)`
> reports `-O2` from the pathdef string. This caused false confidence in the
> bottle's optimisation level during early investigation. The native build is
> immune to this because we call `/usr/bin/clang` directly — the Makefile
> CFLAGS and the actual compiler invocation are the same.

## Homebrew's Default Flags

### What Homebrew uses and why

Homebrew builds *bottles* — pre-compiled binaries that are downloaded and
installed by `brew install`. A bottle must run on **every** Mac that could
install it: M1, M2, M3, M4, and future chips. This forces two conservative
choices:

**`-march=armv8-a`** (lowest common denominator ISA)  
ARMv8-A is the baseline 64-bit ARM spec that every Apple Silicon chip
implements. Using it guarantees the binary won't execute an instruction that
doesn't exist on an older chip. The cost is that the compiler cannot use any
M4-specific instructions, cannot schedule for the M4's pipeline, and cannot
exploit M4's wider SIMD or FMA units.

**`-Os`** (optimise for size, not speed)  
Bottles are served from a CDN and downloaded by every user. Smaller binaries
mean lower bandwidth costs for Homebrew's infrastructure and faster installs
for users. `-Os` disables any optimisation that increases code size — most
notably loop unrolling, aggressive inlining, and function cloning — even when
those would make the code faster.

Both choices are reasonable defaults for a public package manager. They are
the wrong choices for a binary that will only ever run on one specific machine.

### How Homebrew enforces these flags (and why you can't override them)

Homebrew uses a mechanism called **superenv**: a layer of compiler shims that
sits between the formula's `make` invocation and the real compiler.

The relevant file is:
```
/opt/homebrew/Library/Homebrew/Library/Homebrew/extend/os/mac/superenv.rb
```

Inside `setup_build_environment()` (around line 89):

```ruby
ENV["HOMEBREW_OPTIMIZATION_LEVEL"] = "Os"         # always -Os
ENV["HOMEBREW_OPTFLAGS"] = "-march=#{Hardware::CPU.family}"  # always armv8-a on Apple Silicon
```

These are set unconditionally — no formula can override them through normal
means. The shim at:
```
/opt/homebrew/Library/Homebrew/shims/super/cc
```
reads `HOMEBREW_OPTIMIZATION_LEVEL` (line 306) and `HOMEBREW_OPTFLAGS`
(line 307) and injects them into every compile invocation, overwriting
anything the formula or the user set via `CFLAGS`.

### What we tried first (and why it didn't work)

**Attempt 1: `CFLAGS="-O3 -mcpu=native" brew reinstall --build-from-source vim`**

This looks like it should work. It doesn't. The superenv shim runs *after*
the shell exports `CFLAGS`, reads its own `HOMEBREW_OPTIMIZATION_LEVEL` and
`HOMEBREW_OPTFLAGS`, and overwrites the relevant parts of the compile command.
The user-supplied `CFLAGS` are ignored for the flags Homebrew controls.
Verified by checking `strings $(which vim)` after the build — it still showed
`-Os -march=armv8-a` behaviour in the binary.

**Attempt 2: `brew reinstall --build-from-source vim` with a custom tap formula**

Modifying the formula to hardcode `-O3 -mcpu=native` also fails for the same
reason: `setup_build_environment()` runs before any formula code and the shim
overwrites the flags at the compiler call site regardless of what the formula
puts in `CFLAGS`.

**Why `--build-from-source` alone is not enough**

`--build-from-source` only skips downloading the pre-built bottle; it still
runs the full Homebrew build environment including superenv. The result is a
from-source build with the same conservative flags as the bottle. It is slower
to install and produces an identical binary.

### The actual solution: bypass superenv entirely

The only reliable fix is to never let superenv touch the compiler invocation.
The approach in `bootstrap/compile_vim_native.sh`:

1. **`brew fetch --build-from-source vim`** — downloads the source tarball
   without building anything. Homebrew resolves all dependencies and checks
   the tarball hash but does not invoke the shim.

2. **Extract to a `mktemp` directory** — work outside Homebrew's build
   infrastructure entirely.

3. **`CC=/usr/bin/clang ./configure`** — call the *real* compiler, not the
   shim at `/opt/homebrew/Library/Homebrew/shims/super/cc`. Pass the same
   feature flags Homebrew's formula uses (Lua, Python 3, Ruby, Perl, libsodium)
   so the resulting binary has identical capabilities.

4. **`make` with explicit `CFLAGS`** — because we bypassed the shim, our
   `CFLAGS` are the only flags clang sees. No superenv interference.

5. **Copy only the `vim` binary into the existing Cellar** — this preserves
   Homebrew's symlinks, man pages, and metadata. The Cellar binary is made
   writable first (`chmod u+w`) because Homebrew installs bottles read-only.

6. **`brew pin vim`** — prevents `brew upgrade` from silently replacing the
   custom binary with a new bottle.

## Compiler Flags Explained

### Bottle: `-Os`

"Optimise for size." Enables all of `-O2` except transformations that increase
code size. Specifically disables:

- Loop unrolling (each unrolled copy costs space)
- Aggressive function inlining above a low size threshold
- Function cloning / specialisation

The result is a smaller binary that downloads faster. On a loaded server
with many concurrent users, the smaller working set can also improve
instruction-cache efficiency — but on a single-user workstation with a
dedicated hot Vim process this tradeoff is always wrong.

### Bottle: `-march=armv8-a`

Targets the ARMv8-A baseline — the 64-bit ARM spec from 2011. The compiler:

- Assumes only the mandatory ARMv8-A instruction set
- Uses generic instruction scheduling (no knowledge of the M4's pipeline)
- Cannot emit NEON-beyond-baseline, SVE, BFloat16, FMA, or any M-series extension
- Generates code that runs correctly on M1 through M4 and any future chip

### Native: `-O3`

Enables everything in `-O2` plus:

| Transformation | Effect on Vim |
|---|---|
| Auto-vectorisation | Scalar loops over buffer lines / char arrays → NEON SIMD |
| Aggressive loop unrolling | Fewer branch instructions in tight regex scan loops |
| Higher inlining threshold | More call sites in regexp.c / eval.c get inlined |
| Function cloning | Specialised copies for constant arguments (e.g. always-NULL pointer paths eliminated) |

The higher inlining threshold is the most impactful for Vim: the NFA regex
engine and Vimscript interpreter both have deep, tight call chains where
inlining eliminates call overhead and unlocks further optimisations on the
merged code.

### Native: `-mcpu=native`

Detects the exact CPU at compile time. On this machine it resolves to
`apple-m4`. Unlike `-march` (ISA features only) or `-mtune` (scheduling
only), `-mcpu` sets both simultaneously in a consistent way.

What `apple-m4` unlocks over `armv8-a`:

| Extension | Description |
|---|---|
| `+v8.7a` | ARMv8.7-A baseline: `WFXT` (wait-for-event timeout), memory copy/set instructions, xs/hbc |
| `+bf16` | BFloat16 arithmetic — 16-bit float with same 8-bit exponent as float32; used in ML workloads |
| `+i8mm` | 8×8→32-bit integer matrix multiply (`SMMLA`/`UMMLA`); dot-product in one instruction |
| `+sme` / `+sme2` | Scalable Matrix Extension — 2D register tiles for large matrix ops |
| `+fpac` | Pointer Authentication Code instructions for branch-target hardening |

For Vim the primary benefit is **instruction scheduling**: the compiler knows
the M4's pipeline depth, execution unit counts, and instruction latencies, and
arranges instructions to avoid stalls. BFloat16, i8mm, and SME are not
directly used by Vim's code but enabling them allows the compiler to make
better scheduling decisions in adjacent code.

> **ARM flag discipline:** Never mix `-mcpu=native` with an explicit `-march`
> or `-mtune`. `-mcpu` subsumes both; adding either separately causes one to
> partially override the other in ways that are hard to predict and always
> produce worse code than `-mcpu` alone.

### Native: `-ffp-contract=fast`

Allows the compiler to fuse a multiply followed by an add into a single
**FMA (Fused Multiply-Add)** instruction even across statement boundaries.

Standard C: `a * b + c` may generate two instructions (`FMUL` then `FADD`)
to preserve the IEEE 754 intermediate rounding between them.  
With `-ffp-contract=fast`: generates one `FMADD` instruction — one cycle
instead of two, and only one rounding step (more accurate, not less).

On the M4 the FMA units are wide and fast. Vim uses floating-point in
`reltime()` arithmetic and a few internal calculations; none have user-visible
precision requirements, so this is safe. The flag also signals to the backend
that strict IEEE 754 semantics are not required, which occasionally unlocks
additional scheduling freedom in other floating-point adjacent code.

### Native: `-flto`

**Link-Time Optimisation.** Without LTO each `.c` file is compiled
independently. The linker only resolves symbol addresses — it cannot see
inside other translation units. With LTO:

1. Each `.c` file is compiled to **LLVM bitcode** (IR) instead of machine code
2. At link time all bitcode modules are merged into one
3. The optimiser runs across the entire program simultaneously
4. Functions that were opaque across file boundaries become inlineable
5. Dead code that appeared "used" from one file's perspective is eliminated
6. Constants propagate across module boundaries
7. The final machine code is generated once, with full program visibility

For Vim this is most valuable in the **Vimscript interpreter**. The hot
dispatch loop in `eval.c` calls dozens of small helpers spread across
`evalfunc.c`, `typval.c`, `dict.c`, `list.c`, and `strings.c`. Without LTO
these are all opaque `BL` (branch-and-link) instructions. With LTO many get
inlined directly into the dispatch loop, eliminating call overhead and
enabling the backend to see — and optimise — the combined code.

The `vimscript_loop` benchmark shows the largest gain (+23%), which is
consistent with this: a tight interpreter loop where LTO collapses multiple
levels of function call into a handful of instructions.

## Benchmark Details

All benchmarks run inside `vim -Es` (silent batch mode). Timing uses
`reltime()` / `reltimefloat()` inside the Vimscript session. The corpus
file is reloaded from disk with `:e!` before each timed iteration so
buffer state does not carry over between runs.

### `regex_scan` — NFA scan, count only

```vim
%substitute/\w\+//gn
```

Counts all word matches on 100k lines without writing anything. The `n` flag
suppresses substitution — this is pure NFA scan cost.

**NFA (Non-deterministic Finite Automaton):** Vim's regex engine represents a
pattern as a graph of states. At each input character the engine may be in
multiple states simultaneously. This handles back-references and complex
patterns but costs O(states × input_length) in the worst case. The inner
scan loop in `regexp.c` is the primary beneficiary of `-O3` inlining and
`-mcpu=native` scheduling.

**Result:** +22%. Consistent with better instruction scheduling of the NFA
inner loop and higher inlining inside `regexp.c`.

### `regex_replace` — NFA scan + substitution

```vim
%substitute/\<\w\{3,8\}\>/[&]/g
```

Wraps every 3–8 character word in brackets on 100k lines. Adds substitution
overhead on top of the NFA scan: each match triggers a buffer write, shifting
subsequent content. Tests both `regexp.c` and the buffer/substitution path.

**Result:** +15%. Slightly less than pure scan because the substitution I/O
path is less instruction-intensive than the NFA loop.

### `sort` — buffer sort

```vim
%sort
```

Sorts all 100k lines in the buffer. Largely **memory-bandwidth-bound**: the
algorithm spends most time loading and comparing line pointers across a large,
mostly-cold working set. Cache misses dominate; instruction throughput is not
the bottleneck.

**Result:** +4% — within noise for this workload. The native flags give almost
no advantage on a memory-bound task because the CPU spends most cycles waiting
for the cache hierarchy, not executing instructions.

### `vimscript_loop` — interpreter dispatch

```vim
let g:_j = 0
while g:_j < 500000
  let g:_x += g:_j
  let g:_j += 1
endwhile
```

A tight 500k-iteration while loop with integer arithmetic. Each iteration
exercises: variable lookup, integer add, increment, loop-condition check,
branch. No regex, no I/O — pure interpreter overhead.

LTO allows the `do_cmdline()` → `ex_let()` → `eval1()` → `get_var_lval()`
call chain to be partially inlined, reducing per-iteration cost. This is the
clearest LTO signal in the benchmark suite.

**Result:** +23%. The largest gain of any benchmark.

### `regex_ruby` — NFA on structured code

```vim
%substitute/\<\(def\|class\|module\|do\|end\|if\|unless\|rescue\)\>\|:\w\+\|"[^"]*"\|'[^']*'\|#.*$\|[0-9]\+//gn
```

A complex alternation on a 4k-line generated Ruby file (modules, classes,
methods, heredocs, string interpolation, blocks, regex literals, symbols,
DSL-style calls). The pattern mirrors what Vim's Ruby syntax rules match per
line: keywords, symbols, quoted strings, comments, and numbers.

Ruby was chosen over C or Python because its syntax has more NFA states:
heredocs, string interpolation, regex literals, method chains, and operator
symbols all require distinct syntax regions, putting more pressure on the NFA
engine than a language with simpler syntax.

> `syntax sync fromstart` is a no-op in `-Es` batch mode — syntax state is
> computed lazily for display and there is no display. This regex exercises
> the same pattern space directly.

**Result:** +16%. Consistent with NFA scan gains on a richer pattern.

## Observations

- **NFA workloads (+15–22%):** The regex scan loop in `regexp.c` is
  instruction-throughput-bound. Better scheduling (`-mcpu=native`) and higher
  inlining (`-O3`) both contribute.

- **Interpreter gain is the largest (+23%):** The Vimscript eval loop is the
  clearest LTO signal in this suite. Cross-module inlining across `eval.c`,
  `evalfunc.c`, `typval.c`, and friends collapses what was a deep call stack
  into a tight loop.

- **Sort is the outlier (+4%):** Memory-bandwidth-bound workloads do not
  benefit meaningfully from instruction-level optimisations. The bottleneck
  is the cache hierarchy, not the execution units.

- **`-ffp-contract=fast` has no direct measurable impact** on these
  benchmarks — Vim uses floats minimally. Its value is indirect: signalling
  relaxed FP semantics occasionally gives the backend more scheduling freedom
  in adjacent code.

- **The Homebrew defaults are correct for their purpose** (portable,
  size-efficient bottles) and wrong for single-machine use. The superenv
  shim makes it impossible to override them through normal means; bypassing
  it by calling `/usr/bin/clang` directly is the only reliable approach.
